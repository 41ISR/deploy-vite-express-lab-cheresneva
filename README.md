## 1. Подключение к серверу по SSH

**SSH (Secure Shell)** — это протокол для безопасного удалённого управления сервером через терминал. Всё что вы вводите в терминале после подключения — выполняется не на вашем компьютере, а на удалённом сервере.

```bash
ssh root@%ip%
```

Введите пароль, который вам выдали. После успешного входа вы увидите приглашение вида `root@yourserver:~#`.

> **Почему root?** На первом этапе мы входим как суперпользователь, чтобы настроить сервер. Сразу после настройки мы запретим вход под root — это часть защиты сервера.

## 2. Харденинг VPS — защита сервера

Свежий VPS в интернете — лёгкая мишень для автоматических ботов, которые круглосуточно перебирают пароли и ищут уязвимости. Этот раздел закрывает самые очевидные дыры.

> ⚠️ **Выполняйте шаги строго по порядку.** Если заблокируете себя — придётсяначать заного.

---

### 2.1. Обновление системы

Обновим список доступных пакетов и установим свежие версии. Это закроет известные уязвимости в системных компонентах.

```bash
apt update && apt upgrade -y
```

> `apt update` — обновляет список пакетов (что доступно). `apt upgrade -y` — устанавливает обновления без переспросов (`-y` значит "yes" на все вопросы).

### 2.2. Создание пользователя

Никогда не работайте под `root` постоянно. Создадим отдельного пользователя с ограниченными правами.

```bash
# Создаём пользователя (можете заменить deploy на своё имя)
adduser deploy
```

Введите надёжный пароль и нажимайте Enter на остальные вопросы.

```bash
# Даём пользователю право выполнять команды от имени root через sudo
usermod -aG sudo deploy
```

> **Что такое sudo?** Группа `sudo` позволяет пользователю выполнять привилегированные команды с префиксом `sudo`. В отличие от root, каждое привилегированное действие нужно явно запрашивать и подтверждать паролем. Это создаёт защитный барьер от случайных или злонамеренных действий.

### 2.3. 🔑 Генерация и добавление SSH-ключа

> ### ❌ НЕ ДЕЛАТЬ В РАМКАХ ЭТОЙ ЛАБЫ

_Так как SSH траффик заблокирован Ростелекомом даже на выход - мы работаем из под Codespace и у нас нет хорошего метода для работы с SSH ключами. В рамках когда вы разворачиваете сервис со своего компьютера - этот шаг один из главных_

**SSH-ключ** — это пара файлов: **приватный** (хранится у вас на компьютере, никому не передавать) и **публичный** (кладётся на сервер). Сервер пускает только того, у кого есть приватный ключ, соответствующий публичному. Это надёжнее пароля — перебрать ключ практически невозможно.

**На вашем локальном компьютере** (не на сервере!):

```bash
# Генерируем пару ключей
# ed25519 — современный алгоритм, компактный и надёжный
ssh-keygen -t ed25519 -C "your_email@example.com"
```

Ключи сохранятся в `~/.ssh/id_ed25519` (приватный) и `~/.ssh/id_ed25519.pub` (публичный). Никогда не передавайте приватный ключ никому и никуда.

_`~/` - обычно директория вашего пользователя или документы, однако `.ssh` чаще всего скрытая папка. На линуксовых системах это ваша папка внутри `home` и она там точно скрыта, так как начинается с `.`_

```bash
# Копируем публичный ключ на сервер
# Команда сама добавит его в ~/.ssh/authorized_keys на сервере
ssh-copy-id -i ~/.ssh/id_ed25519.pub deploy@yourname.example.com
```

После этого подключение работает без пароля:

```bash
ssh deploy@yourname.example.com
```

**На сервере** — проверяем, что ключ добавился:

```bash
cat ~/.ssh/authorized_keys
# Должна появиться строка, начинающаяся с "ssh-ed25519 ..."
```

### 2.4. Смена порта SSH

По умолчанию SSH работает на порту **22** — именно его сканируют боты в первую очередь. Смена порта не является абсолютной защитой, но существенно снижает количество автоматических атак и мусора в логах.

```bash
sudo nano /etc/ssh/sshd_config
```

Найдите строку `#Port 22` и замените на:

```
Port 2222
```

> Выберите порт от 1024 до 65535. Мы используем `2222` как пример.

Сохраните файл: `Ctrl+O` → Enter → `Ctrl+X`.

### 2.5. Отключение входа под root по SSH

> ⚠️ **Не закрывайте текущее соединение!** Откройте второй терминал и убедитесь, что можете войти под `deploy` на новом порту, прежде чем продолжить.

В том же файле `/etc/ssh/sshd_config` найдите и измените:

```
PermitRootLogin no
```

> **Почему отключаем root?** Имя пользователя `root` известно всем — боты в первую очередь атакуют именно его. Отключив вход root по SSH, мы убираем этот вектор атаки. Пользователь `deploy` с sudo требует знания ещё и имени пользователя.

> Строку `PasswordAuthentication no` трогаем **только если** настроили SSH-ключи. Без ключей это заблокирует все входы.

Перезапускаем SSH-сервер, чтобы изменения вступили в силу:

```bash
sudo systemctl daemon-reload
sudo systemctl restart ssh
sudo systemctl restart sshd
```

_На Ubuntu < 22 и почти всех других системах это делается командой_

```bash
sudo systemctl restart sshd
```

Теперь подключение к серверу выглядит так:

```bash
ssh -p 2222 deploy@%ip%
```

### 2.6. Настройка файрвола UFW

**UFW (Uncomplicated Firewall)** — простой инструмент управления сетевым фильтром Linux. По умолчанию на VPS открыты все порты — любой желающий может попытаться подключиться к любому сервису. Файрвол закрывает всё лишнее.

```bash
# Разрешаем наш новый SSH-порт — ОБЯЗАТЕЛЬНО до включения файрвола!
# Иначе заблокируем себя
sudo ufw allow 2222/tcp

# Разрешаем HTTP и HTTPS — для работы Swag
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# Включаем файрвол
sudo ufw enable
```

На вопрос `Proceed with operation (y|n)?` отвечаем `y`.

```bash
# Проверяем статус — убеждаемся, что нужные порты открыты
sudo ufw status verbose
```

Вывод должен выглядеть примерно так:

```
Status: active
To                         Action      From
--                         ------      ----
2222/tcp                   ALLOW IN    Anywhere
80/tcp                     ALLOW IN    Anywhere
443/tcp                    ALLOW IN    Anywhere
```

> **Что происходит:** UFW по умолчанию блокирует все входящие соединения кроме тех, что мы явно разрешили. Порт 22 мы намеренно **не** разрешаем — SSH теперь работает только на 2222. Порты 80 и 443 нужны для Swag (HTTP и HTTPS).

## 3. Установка Docker и Docker Compose

**Docker** — платформа для запуска приложений в изолированных **контейнерах**. Контейнер содержит всё необходимое для работы приложения: код, зависимости, конфиги — и работает одинаково на любом сервере независимо от того, что на нём установлено.

**Docker Compose** — инструмент для запуска нескольких связанных контейнеров как единого приложения.

```bash
# Переключаемся на пользователя deploy (если ещё под root)
su - deploy

# Устанавливаем зависимости для добавления репозитория
sudo apt-get install -y ca-certificates curl

# Создаём папку для ключей и добавляем официальный GPG-ключ Docker
# GPG-ключ нужен для проверки подлинности скачиваемых пакетов
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# Добавляем официальный репозиторий Docker в apt
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Обновляем список пакетов и устанавливаем Docker
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

> **Почему не `apt install docker`?** В стандартном репозитории Ubuntu лежит устаревшая версия Docker. Мы подключаем официальный репозиторий Docker Inc., чтобы получить актуальную версию.

```bash
# Добавляем в группу docker (чтобы мог управлять контейнерами)
usermod -aG docker deploy
# Применяем изменение группы без перелогина
newgrp docker

# Проверяем установку
docker --version
docker compose version
```

## 5. Создание Dockerfile для бэкенда

**Dockerfile** — текстовый файл с пошаговыми инструкциями для сборки Docker-образа. Образ — это неизменяемый шаблон, из которого запускаются контейнеры.

Создайте файл `Dockerfile` в папке бэкенда:

```bash
nano backend/Dockerfile
```

```dockerfile
# ─────────────────────────────────────────────
# Этап 1: Установка зависимостей (builder)
# ─────────────────────────────────────────────

# Берём официальный образ Node.js 22
FROM node:22 AS builder

# Устанавливаем рабочую директорию внутри контейнера.
# Все последующие команды будут выполняться относительно неё.
WORKDIR /app

# Копируем файлы зависимостей ПЕРВЫМИ — до копирования остального кода.
# Docker кэширует каждый слой. Если package.json не изменился,
# npm install при следующей сборке будет взят из кэша — это экономит время.
COPY package*.json ./

# Устанавливаем только production-зависимости.
# npm ci — строго по package-lock.json, гарантирует воспроизводимость.
RUN npm ci --only=production

# ─────────────────────────────────────────────
# Этап 2: Финальный образ
# ─────────────────────────────────────────────

FROM node:22

WORKDIR /app

# Копируем установленные зависимости из этапа builder.
# В финальный образ не попадут кэши npm и инструменты сборки.
COPY --from=builder /app/node_modules ./node_modules

# Копируем исходный код приложения
COPY . .

# Документируем, что приложение слушает порт 3000.
# EXPOSE не открывает порт наружу — это лишь документация.
EXPOSE 3001

# Команда, которая будет выполнена при старте контейнера.
# Замените src/index.js на реальный путь к точке входа вашего приложения.
CMD ["node", "src/index.js"]
```

> **Почему многоэтапная сборка?** Она позволяет не включать в финальный образ инструменты сборки и лишние файлы. В итоге образ меньше весит и имеет меньшую поверхность атаки.

## 6. Настройка переменных окружения (.env)

Секреты — пароли, API-ключи, строки подключения к БД — **нельзя хранить в репозитории**. Для этого используется файл `.env`, который создаётся вручную прямо на сервере.

```bash
nano .env
```

Пример содержимого:

```env
# Порт бэкенда
PORT=3001

# Окружение
NODE_ENV=production

# Пример — строка подключения к БД (если нужна)
# DATABASE_URL=postgresql://user:password@localhost:5432/mydb
```

> **Важно:** Убедитесь, что `.env` добавлен в `.gitignore` в вашем репозитории, иначе секреты попадут в публичный доступ на GitHub.

## 7. Настройка Swag (NGINX + SSL)

**Swag** (Secure Web Application Gateway) от LinuxServer.io — готовый Docker-контейнер, который объединяет:
- **NGINX** — веб-сервер и обратный прокси
- **Certbot** — автоматическое получение и обновление SSL-сертификата от Let's Encrypt

### Как это работает

```
Браузер → [yourname.example.com:443 HTTPS]
              ↓
           [Swag / NGINX]
              ├── /api/* → проксирует на backend:3000
              └── /*     → отдаёт статику фронтенда (dist/)
```

Пользователь всегда общается со Swag по HTTPS. Swag сам решает, что отдать — статику или ответ бэкенда.

### Конфиг NGINX

Создайте папку и файл конфигурации:

```bash
mkdir -p swag/nginx/site-confs
nano swag/nginx/site-confs/default.conf
```

```nginx
server {
    listen 443 ssl;
    listen [::]:443 ssl;

    server_name %yourname.example.com%;

    # SSL-сертификаты Swag получает автоматически и управляет ими сам
    include /config/nginx/ssl.conf;

    # ───────────────────────────────────────────
    # Фронтенд — статические файлы
    # ───────────────────────────────────────────

    # Корневая папка, откуда отдаются файлы фронтенда
    root /config/www;
    index index.html;

    # Для SPA (React, Vue): NGINX ищет файл, потом папку,
    # и если ничего не нашёл — отдаёт index.html.
    # Так работает клиентская маршрутизация: роуты вида /about
    # обрабатываются JavaScript в браузере, а не сервером.
    location / {
        try_files $uri $uri/ /index.html;
    }

    # ───────────────────────────────────────────
    # Бэкенд — проксирование API-запросов
    # ───────────────────────────────────────────

    # Все запросы на /api/... перенаправляются на бэкенд.
    # "backend" — имя сервиса в docker-compose, он резолвится
    # как hostname внутри Docker-сети.
    location /api/ {
        proxy_pass http://backend:3001;

        # Передаём оригинальные заголовки запроса, чтобы бэкенд
        # знал реальный IP клиента, а не IP Docker-сети
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

# Перенаправляем все HTTP-запросы на HTTPS
server {
    listen 80;
    listen [::]:80;

    server_name yourname.example.com;
    return 301 https://$host$request_uri;
}
```

## 8. Настройка Docker Compose

Создайте файл `docker-compose.yml` в корне проекта:

```bash
nano docker-compose.yml
```

```yaml
version: "3.8"

services:

  # ─────────────────────────────────────────────
  # Swag — обратный прокси + автоматический SSL
  # ─────────────────────────────────────────────
  swag:
    image: lscr.io/linuxserver/swag:latest
    container_name: swag
    cap_add:
      - NET_ADMIN  # Необходимо для управления сетевыми настройками внутри контейнера
    environment:
      - PUID=1000             # ID пользователя системы (узнать командой: id -u)
      - PGID=1000             # ID группы (узнать: id -g)
      - TZ=Europe/Moscow      # Временная зона — для корректных меток времени в логах
      - URL=yourname.example.com  # ВАШ поддомен — для получения SSL-сертификата
      - VALIDATION=http       # Метод подтверждения домена для Let's Encrypt
    volumes:
      # Папка с конфигами Swag: сертификаты, NGINX-конфиги.
      # Сертификаты сохраняются на диске и не теряются при перезапуске контейнера.
      - ./swag:/config
      # Папка со статическими файлами фронтенда.
      # :ro — монтируем только для чтения (контейнер не может изменить файлы)
      - ./frontend/dist:/config/www:ro
    ports:
      - "443:443"  # HTTPS — пробрасываем из контейнера наружу
      - "80:80"    # HTTP — нужен для редиректа и проверки домена Let's Encrypt
    restart: unless-stopped
    networks:
      - app-network

  # ─────────────────────────────────────────────
  # Backend — Node.js + Express
  # ─────────────────────────────────────────────
  backend:
    build:
      context: ./backend   # Папка, где находится Dockerfile
      dockerfile: Dockerfile
    container_name: backend
    env_file:
      - .env               # Docker передаёт все переменные из .env в контейнер
    expose:
      - "3001"
      # expose ≠ ports: порт доступен только внутри Docker-сети (для Swag),
      # но НЕ снаружи. Бэкенд не торчит в открытый интернет напрямую.
    restart: unless-stopped
    networks:
      - app-network

# Docker-сеть — виртуальная локальная сеть для контейнеров.
# Контейнеры в одной сети обращаются друг к другу по имени сервиса
# (например, http://backend:3001), как по доменному имени.
networks:
  app-network:
    driver: bridge
```

## 10. Запуск приложения

```bash
# Убедитесь, что вы в корне проекта
cd ~/your-repo

# Собираем образы и запускаем все контейнеры в фоновом режиме
# --build: пересобрать образы перед запуском
# -d: detached mode — контейнеры работают в фоне, терминал свободен
docker compose up -d --build
```

**Что происходит при первом запуске:**
1. Docker скачивает базовый образ `node:22` и образ Swag
2. Docker собирает образ бэкенда по вашему Dockerfile
3. Контейнеры стартуют и подключаются к общей сети `app-network`
4. Swag обращается к серверам Let's Encrypt и получает SSL-сертификат для вашего домена
5. NGINX стартует и начинает принимать запросы

Первый запуск занимает **1–2 минуты** (получение сертификата).

```bash
# Проверяем статус контейнеров
docker compose ps
```

Все сервисы должны быть в статусе `running`. Откройте браузер: `https://yourname.example.com`.

## 11. Полезные команды для отладки

```bash
# Статус всех контейнеров
docker compose ps

# Логи всех контейнеров в реальном времени (Ctrl+C для выхода)
docker compose logs -f

# Логи конкретного сервиса
docker compose logs -f backend
docker compose logs -f swag

# Перезапустить один сервис (без пересборки)
docker compose restart backend

# Пересобрать и перезапустить после изменений в коде
git pull
docker compose up -d --build

# Остановить все контейнеры
docker compose down

# Остановить и удалить тома (осторожно — удаляет данные!)
docker compose down -v

# Зайти внутрь работающего контейнера для ручной отладки
docker exec -it backend sh

# Мониторинг использования CPU и памяти в реальном времени
docker stats
```
